## 1. 문서 목적

이 보고서는 **개발 PC에서 MagicSquare 4×4 테스트를 실제로 돌리기 위한 절차**를 담는다. 아래 순서대로 진행하면 Boundary / Domain / E2E 테스트를 재현할 수 있다.

---

## 2. 테스트 조건 (환경·도구)

| 구분 | 요구 사항 |
|------|-----------|
| Python | **3.10 이상** (`pyproject.toml`의 `requires-python`) |
| 테스트 러너 | **pytest** 8.x 이상 (선택 의존성 `dev`) |
| 작업 디렉터리 | 저장소 **루트** (`pyproject.toml`이 있는 폴더) |
| OS | Windows / macOS / Linux (경로만 본인 환경에 맞게 바꿀 것) |
| 가상 환경 | **권장**: `venv`로 프로젝트별 Python·pip 격리 ([README 설치](../README.md) 참고) |

**pytest 설정** (`pyproject.toml` 기준):

- `pythonpath = ["src"]` → 패키지 `magicsquare` 를 `src`에서 로드
- `testpaths = ["tests"]` → 기본 수집 경로는 `tests/`

## 3. 가상 환경에서 실행하기 (권장)

의존성을 시스템 Python과 섞지 않으려면 **먼저 venv를 만들고 활성화**한 뒤, 아래 체크리스트의 `pip` / `pytest` 는 모두 그 터미널에서 실행한다.

### 3.1 생성·활성화

**Windows PowerShell** (저장소 경로는 본인 PC에 맞게 수정):

```powershell
Set-Location C:\DEV\MagicSquare
python -m venv .venv
.\.venv\Scripts\Activate.ps1
```

- `Activate.ps1` 실행이 막히면: `Set-ExecutionPolicy -ExecutionPolicy RemoteSigned -Scope CurrentUser`

**Windows cmd:**

```bat
cd C:\DEV\MagicSquare
python -m venv .venv
.venv\Scripts\activate.bat
```

**macOS / Linux:**

```bash
cd /path/to/MagicSquare
python3 -m venv .venv
source .venv/bin/activate
```

활성화에 성공하면 프롬프트 앞에 `(.venv)` 가 붙는 경우가 많다. 종료 시 같은 터미널에서 `deactivate`.

### 3.2 가상 환경 안에서 패키지 설치

활성화된 상태에서 저장소 **루트**에서:

```bash
python -m pip install -U pip
pip install -e ".[dev]"
```

이때 설치되는 `pytest` 는 **`.venv` 안**에만 들어간다.

## 4. 전제 조건 (실행 전 체크리스트)

아래를 **순서대로** 확인한다.

1. [ ] 터미널에서 `python --version` → **3.10+** 인지 확인한다. (**가상 환경 활성화 후**에 확인하면, venv가 쓰는 Python 버전이 나온다.)
2. [ ] 저장소 루트로 이동한다.  
   - PowerShell 예: `Set-Location C:\DEV\MagicSquare`
3. [ ] **권장**: §3에 따라 가상 환경을 만들고 **활성화**한다.
4. [ ] (최초 1회 또는 의존성 갱신 후) §3.2대로 편집 가능 설치:
   ```bash
   pip install -e ".[dev]"
   ```
5. [ ] 테스트 수집이 되는지 확인한다:
   ```bash
   python -m pytest tests --collect-only -q
   ```
   - 기대: **`22 tests collected`** (현재 저장소 기준). 개수가 다르면 브랜치·파일 상태를 확인한다.
6. [ ] PRD **§5** `code` / `message` 문자열이 테스트 기대값과 일치하는지 코드 리뷰 시 참고한다(불일치 시 테스트 실패 원인의 대부분).

## 5. 테스트 실행 절차

가상 환경을 쓰는 경우 **§3.2까지 끝낸 같은 터미널**에서 실행한다.

### 5.1 전체 테스트 (권장 기본)

저장소 루트에서:

```bash
python -m pytest tests
```

**축약 출력**이 필요하면:

```bash
python -m pytest tests -q
```

**기대(성공 시)**:

- 종료 코드 **0**
- 요약에 **`passed`** 만 있고, `failed` / `errors` 가 없음
- 최근 확인 기준: **22 passed** (약 0.1초 전후)

실패 시 첫 번째 실패에서 멈추려면:

```bash
python -m pytest tests -x
```

상세 스택이 필요하면:

```bash
python -m pytest tests -vv --tb=long
```

---

### 5.2 트랙(Dual-Track)별 실행

| 트랙 | 명령 | 검증 범위(요지) |
|------|------|-----------------|
| Boundary | `python -m pytest tests/boundary -v` | FR-01, 오류 우선순위, Domain 미호출 |
| Domain | `python -m pytest tests/domain -v` | FR-02~05, TQ-NOSOL, 검증기·리졸버 |
| E2E | `python -m pytest tests/e2e -v` | 공개 API, AC-02~04, 오류 dict / 성공 list |

---

### 5.3 단일 파일·단일 케이스 실행

**파일 단위:**

```bash
python -m pytest tests/e2e/test_api_e2e.py -v
```

**함수 단위 (이름 예시):**

```bash
python -m pytest tests/e2e/test_api_e2e.py::test_e2e_ok_01_returns_int6_for_valid_solvable_grid -v
```

```bash
python -m pytest tests/domain/test_nosol_preproof.py::test_tq_nosol_both_candidate_placements_are_not_magic -v
```

## 7. 성공 / 실패 판단 기준

### 7.1 성공

- `python -m pytest tests` 가 **오류 없이 끝나고**, 보고되는 **`passed` 수가 수집된 테스트 수와 같다**.
- Boundary 테스트가 Green이면 **AC-01** 및 **무효 입력 시 Domain 미호출**이 유지되는 쪽으로 동작 중인 것이다.
- E2E가 Green이면 **응답 형식·`NO_SOLUTION`·결정론·비변형**이 PRD 방향과 맞는 것이다.

### 7.2 실패 (조치 방향)

| 증상 | 우선 확인 |
|------|-----------|
| `ImportError` / `ModuleNotFoundError: magicsquare` | 루트에서 실행했는지, **가상 환경 활성화** 후 `pip install -e ".[dev]"` 했는지, `pythonpath` 설정 여부 |
| `AssertionError` on `code` / `message` | PRD **§5** 문구와 구현·테스트 기대값 **완전 일치** 여부 |
| Domain 관련 실패 | FR-05 시도 순서, FR-04 합 34·10개 선 정의 |
| `collect` 단계에서 테스트 수 감소 | `tests/` 파일 삭제·이름 변경, `pytest` 설정 변경 여부 |

## 8. 커버리지 (선택, PRD NFR-01·02)

`pyproject.toml`의 `dev` 세트에는 **pytest-cov** 가 포함되어 있지 않다. NFR을 수치로 맞추려면 예:

```bash
pip install pytest-cov
python -m pytest tests --cov=magicsquare --cov-report=term-missing
```

임계치(PR **NFR-01** Domain ≥95%, **NFR-02** Boundary ≥85%)는 팀에서 `pyproject` 또는 CI 스크립트에 `--cov-fail-under` 등으로 붙이면 된다.

## 9. 특별 절차 (기록·회귀·보고)

1. **결과 기록**: 릴리스·PR 단위로 `python -m pytest tests -v` 로그를 저장하거나 CI 아티팩트에 남긴다.
2. **실패 보고**: 실패한 `테스트 파일::함수명`, 입력 fixture 또는 최소 재현 격자, 기대 `code`/`message` 또는 기대 `list` 길이를 정리하고 PRD **FR·AC** 줄을 명시한다.
3. **`NO_SOLUTION`**: 입력 검증(`INVALID_*`)과 혼동하지 말 것. E2E에서 `NO_SOLUTION` 은 **유효한 격자**에 대한 응답이다(PRD §5 구분).
4. **추적성**: 새 테스트 추가 시 [`docs/6.TODO_MagicSquare_4x4_Checklist.md`](../docs/6.TODO_MagicSquare_4x4_Checklist.md) 및 PRD §8.4 시나리오와 맞춘다.
