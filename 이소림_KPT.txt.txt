이소림 KPT

K  / Keep (유지할 만한 것)

- PRD 한 장에 요구 사항, 오류 문구, 우선 순위를 모아 볼 수 있어 어떤 것을 목표로 할지가 여러번의 프롬프팅에도 흔들리지 않았음
- 로직과 테스트 폴더 분류 아키텍쳐링 : 리팩토링, 기능 추가 시 어디를 확인해야할지가 명확해짐
- 선 테스트 후 반영 : Green 단계 유지한채 문서, gui, 작은 함수 단위로 리팩토링 가능
- 보고서, 프롬프트 정리 : 히스토리 파악 가능
- 가상환경 및 실행 방법 작성 : 타 개발자도 바로 실행 가능


P / Problem(아쉬웠던 점)

- main 브렌치와 작업 브렌치가 어긋나면 기준 브렌치 혼동 가능
- GUI는 수동으로 눌러봐야 확인이 가능하여, CI만으로는 완벽한 자동검증 불가능
- 쓸데없이 추가되는 코드 및 모듈 사용 가능 : gui 확인을 위해 tinker 사용하게 되었는데 gui 개발을 위한 다양하고 더 쉬운 옵션이 있는지 검증 로직 필요


T / Try (다음에 시도할것)

- GUI 수동 체크를 위한 개발자 자주 검증 사항에 대한 TC를 ReadMe 파일에 기재하기
- PRD나 부록에 "테스트 함수 <-> FR/AC " 작은 표 1개를 만들어, 변경 시 갱신
- 작업 단계 별 리뷰 꼼꼼히 하여, 생성형 AI를 통해 산출물 확인에 급급하기 보다 프로젝트에 더 알맞는 선택을 할 수 있도록 검수
