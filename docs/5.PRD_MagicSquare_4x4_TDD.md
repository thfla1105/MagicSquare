# PRD: MagicSquare 4x4 (Dual-Track TDD)

작성일: 2026-04-27  
최종 수정: 2026-04-28 — Dual Track UI + Logic · MLOps 역할, **§2.4 tkinter 로컬 GUI** 반영  

문서 목적: 4x4 Magic Square 문제를 **Boundary(API·입력 계약)** 와 **Domain(순수 로직)** 의 이중 트랙 TDD로 구현하기 위한 단일 기준 문서. 슬라이드 자료의 **UX Contract / Logic Rule** 언어와 **3단 매핑**을 본 PRD의 FR·AC·테스트 전략에 정렬한다.

## 1. 배경 및 목표

부분적으로 비어 있는 4x4 격자(빈칸은 `0`)를 받아, 두 빈칸에 값을 배치하여 완성 가능한지 판단한다.

- 완성 가능: `int[6]` 반환 (`[r1,c1,n1,r2,c2,n2]`, 1-index)
- 완성 불가: 표준 오류 계약 `NO_SOLUTION` 반환
- 계약 위반 입력: 표준 `code`/`message` 반환

본 PRD는 기존 진행 문서(제안사항/현황 보고)의 내용을 반영하여 구현·테스트·검증 기준을 통합한다.

## 2. 범위

### 2.1 In Scope

- 4x4 고정 격자
- 값 도메인: `0` 또는 `1..16`
- 빈칸 개수: 정확히 2개
- 10개 선(4행, 4열, 주/부대각선) 합이 모두 34인 경우만 정답
- 두 배치 시도(낮은 수→큰 수, 큰 수→낮은 수)

### 2.2 Out of Scope

- N×N 일반화
- **웹·SPA·원격 API 서버, DB** 등 브라우저/서버 중심 배포
- 임의 퍼즐 생성기

### 2.3 Dual Track과 UI 범위

- **도메인·계약의 기준**은 여전히 `solve_magic_square`(**§6**) 이다. **§2.4** 로컬 GUI는 이 API를 호출하는 **표현층**일 뿐이며, 오류 문구·우선순위는 Boundary와 동일해야 한다.
- Dual Track 관점에서 **경계(Boundary)** 가 정의하는 **API 계약**(성공 `list` vs 실패 `dict`, `code`/`message`) 은 GUI 없이도 성립하고, GUI가 있으면 **동일 계약**이 사용자에게 보이도록 맞춘다.
- **React 등 웹 프론트**를 둘 경우, **동일 AC** 를 UX Contract 언어로 옮기고, **로직 단위 테스트는 DOM·표현층 세부를 몰라도 된다**는 원칙을 유지한다.

### 2.4 로컬 데모 GUI (In Scope)

- Python 표준 **`tkinter`** 기반 데스크톱 화면: 4×4 입력, 풀이, 결과(성공 시 격자 반영 또는 메시지, 실패 시 `code`/`message`).
- 구현 위치: `src/magicsquare/gui/` — 실행은 `magicsquare-gui` 콘솔 엔트리 또는 `python -m magicsquare.gui.tk_app` ( `README` 참고).

## 3. 용어 정의

- **Boundary**: 입력 계약 검증 및 외부 응답 형식 책임
- **Domain**: 순수 해 결정 로직 책임
- **Control(UseCase)**: Boundary 통과 시 Domain 호출 및 결과 매핑
- **Logic Track(로직 트랙)**: Domain(·필요 시 Control) 중심. **Logic Rule** 언어: 허용/거부, 유지/중단(검색·해 탐색), 반환/차단(해·`NO_SOLUTION`), 계산(빈칸·누락 수·10선 합) 등.
- **UI/Boundary Track(경계·UI 트랙)**: GUI가 없을 때는 **API·Boundary** 가 “사용자(클라이언트)에 보이는 결과”를 담당. **UX Contract** 언어: 보인다/안 보인다, 가능/불가, 활성/비활성(해당 시), 포함/미포함(오류 문구·형식) 등.
- **독립성**: UI(또는 API Boundary) 테스트는 **비지니스 수식**을 몰라도 되고, Logic 테스트는 **React/HTTP** 를 몰라도 된다. **GREEN·REFACTOR** 는 가능한 한 **동시에** 품질을 맞춘다.

## 4. 기능 요구사항 (FR)

### FR-01 입력 검증

입력 `grid`는 다음 계약을 만족해야 한다.

1. 4x4 크기
2. 각 셀 값이 `0` 또는 `1..16`
3. `0`이 정확히 2개
4. `0` 제외 `1..16` 중복 없음

위반 시 즉시 오류 반환하고 Domain 해 결정 로직은 호출하지 않는다. (Logic Rule: **중단**·**차단**)

### FR-02 빈칸 탐색

빈칸 2개 좌표는 행 우선(row-major)으로 탐색한다.

### FR-03 누락 수 계산

`1..16`에서 현재 격자에 존재하는 수(0 제외)를 제거해 누락된 두 수 `{a,b}`를 구한다.

### FR-04 완성 판정

`0`이 없는 완전 격자에 대해 아래 10개 합이 모두 34이면 참:

- 4개 행
- 4개 열
- 주대각선
- 부대각선

### FR-05 해 결정 및 반환

- 누락 두 수에 대해 최대 두 번 시도:
  - 시도1: 작은 수를 첫 빈칸, 큰 수를 둘째 빈칸
  - 시도2: 반대로 배치
- 성공 시 `int[6]` 반환
- 두 시도 모두 실패 시 `NO_SOLUTION` 반환
- 호출자가 전달한 `grid`는 변경하지 않는다(NFR-04)

## 5. 오류 계약 (code/message)

문자열은 아래와 정확히 일치해야 한다.

| code | message |
|---|---|
| `INVALID_SIZE` | `Grid must be 4x4.` |
| `INVALID_VALUE_RANGE` | `Each cell must be 0 or 1..16.` |
| `INVALID_ZERO_COUNT` | `There must be exactly two zeros (empty cells).` |
| `DUPLICATE_VALUES` | `Values 1..16 must not duplicate (excluding zeros).` |
| `NO_SOLUTION` | `No placement makes a 4x4 magic square with magic sum 34.` |

**입력 검증(FR-01) 우선순위**(유효하지 않은 입력에 어떤 코드가 나오는지):

`INVALID_SIZE` → `INVALID_VALUE_RANGE` → `INVALID_ZERO_COUNT` → `DUPLICATE_VALUES`

**구분**: `NO_SOLUTION` 은 **입력 형식 위반이 아니라**, 유효 입력에 대해 Domain의 두 배치 시도가 모두 실패했을 때 반환하는 **해 결정 결과**이다. 위 우선순위 목록에는 **포함하지 않는다.**

## 6. API 계약

### 6.1 입력

- 타입: `list[list[int]]` (4x4)

### 6.2 출력

- 성공: `list[int]` 길이 6
  - `[r1,c1,n1,r2,c2,n2]` (좌표 1-index)
- 실패: `dict[str, str]`
  - `{"code": "...", "message": "..."}`

## 7. 비기능 요구사항 (NFR)

- NFR-01: Domain 라인 커버리지 >= 95%
- NFR-02: Boundary 라인 커버리지 >= 85%
- NFR-03: 동일 입력에 동일 출력(결정론)
- NFR-04: 입력 `grid` 비변형 보장

## 7.5 MLOps (선택·역할 분리)

- **Ground truth**: PRD **FR-04·FR-05** 및 이에 따른 **AC·골든/회귀 테스트**가 “맞음/틀림”의 기준이며, **최종 정답·검증 로직을 ML로 대체하지 않는다** (교육·실험용 보조 모델은 예외적으로 별도 계약).
- **이 저장소의 Dual Track TDD**: 녹색 막대의 근거는 **결정론적 Boundary·Domain·E2E** 테스트에 둔다.
- **MLOps에 두기 좋은 것**(프로젝트에서 모델·데이터 파이프라인을 쓸 경우): 실험·데이터셋·모델 **버전**, **재현성**, 배포 파이프라인; CI의 **커버리지 게이트**(NFR-01·02)와 함께 **품질 게이트**로 운용 가능.
- **PRD §2.2** (GUI/DB 등) 범위는 MLOps **스택 선택**과 별개이나, **DB/서빙**을 Scope에 넣는 경우 본 절을 **제품·배포** 문서와 맞춰 갱신한다.

## 8. 테스트 전략 (Dual-Track TDD)

### 8.0 핵심 원리

- **UI(또는 API) 쪽**은 설계·계약으로 RED, **로직 쪽**은 규칙으로 RED — **GREEN·REFACTOR** 는 양 트랙을 함께 맞춘다.
- 아래 **§8.4** 매핑 표에 **판단(Decision)** 이 있는 시나리오만 테스트 케이스로 내린다. 동사로 끝나지 않는 모호한 할 일 목록은 테스트 변환 대상에서 제외한다.

### 8.1 Boundary Track

- 계약 오류 코드/문구/우선순위 검증
- 무효 입력 시 Domain 미호출 검증
- UX Contract 관점: 호출자에게 **어떤 오류/성공 형태가 보이는지** (dict vs `list` 6요소)

### 8.2 Domain Track

- 빈칸 탐색, 누락 수 계산, 판정 로직 단위 검증
- 두 배치 시도 로직 검증
- `TQ-NOSOL` 두 시도 모두 실패 자동 증명
- Logic Rule 관점: 해 **반환**, **`NO_SOLUTION` 거부**, **계산**·**유지**(입력 비변형) — **UI·API는 몰라도 됨**

### 8.3 E2E

- 정상 해결(`E2E-OK`)
- 해 없음(`E2E-NO`)
- 입력 오류(`E2E-ERR` 계열)
- 결정론/입력 비변형

### 8.4 시나리오 · UX Contract(또는 API) · Logic Rule 매핑

| 시나리오 (난이도·우선순위) | UX / Boundary (무GUI면 API 계약) | Logic Rule |
|:---|:---|:---|
| 입력 검증 실패 (크기·값·0 개수·중복) | 해당 **code/message** 가 반환된다 · 해 형태는 반환되지 않는다 | **차단한다** · Domain **호출 안 함**(중단) |
| 유효 입력·두 시도 내 해 존재 | **`int[6]` 성공** 이 반환된다 | **반환한다**(해) |
| 유효 입력·두 시도 모두 실패 | **`NO_SOLUTION`** 계약이 반환된다 | **거부한다**(배치 불가) |
| 동일 입력 재호출 (NFR-03) | **동일 출력** 이 보인다 | **결정론 유지** |

### 8.5 UI 트랙

- **§2.4** 로컬 **tkinter** GUI는 수동으로 **§8.4** 시나리오를 확인할 수 있다(자동 E2E UI 테스트는 선택).
- **웹 UI** 를 추가할 경우: 동일 시나리오를 **보인다/안 보인다**, 입력 **불가/오류 표시** 등으로 테스트하며, **§8.4** Logic 열과 1:1 대응을 유지한다.

## 9. 수용 기준 (AC)

- AC-01: 유효하지 않은 입력은 우선순위 규칙대로 오류 반환
- AC-02: 유효 입력에 대해 두 시도 내 해가 있으면 `int[6]` 반환
- AC-03: 두 시도 모두 실패하면 `NO_SOLUTION` 반환
- AC-04: 모든 결과는 결정론적이며 입력은 변경되지 않음
- AC-05: 테스트 통과 + NFR 커버리지 기준 충족
