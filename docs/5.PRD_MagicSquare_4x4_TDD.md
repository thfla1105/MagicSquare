# PRD: MagicSquare 4x4 (Dual-Track TDD)

작성일: 2026-04-27  
문서 목적: 4x4 Magic Square 문제를 Boundary/Domain 이중 트랙 TDD로 구현하기 위한 단일 기준 문서

## 1. 배경 및 목표

부분적으로 비어 있는 4x4 격자(빈칸은 `0`)를 받아, 두 빈칸에 값을 배치하여 완성 가능한지 판단한다.

- 완성 가능: `int[6]` 반환 (`[r1,c1,n1,r2,c2,n2]`, 1-index)
- 완성 불가: 표준 오류 계약 `NO_SOLUTION` 반환
- 계약 위반 입력: 표준 `code`/`message` 반환

본 PRD는 기존 진행 문서(제안사항/현황 보고)의 내용을 반영하여 구현·테스트·검증 기준을 통합한다.

## 2. 범위

### 2.1 In Scope

- 4x4 고정 격자
- 값 도메인: `0` 또는 `1..16`
- 빈칸 개수: 정확히 2개
- 10개 선(4행, 4열, 주/부대각선) 합이 모두 34인 경우만 정답
- 두 배치 시도(낮은 수→큰 수, 큰 수→낮은 수)

### 2.2 Out of Scope

- N×N 일반화
- GUI/웹/DB
- 임의 퍼즐 생성기

## 3. 용어 정의

- **Boundary**: 입력 계약 검증 및 외부 응답 형식 책임
- **Domain**: 순수 해 결정 로직 책임
- **Control(UseCase)**: Boundary 통과 시 Domain 호출 및 결과 매핑

## 4. 기능 요구사항 (FR)

### FR-01 입력 검증

입력 `grid`는 다음 계약을 만족해야 한다.

1. 4x4 크기
2. 각 셀 값이 `0` 또는 `1..16`
3. `0`이 정확히 2개
4. `0` 제외 `1..16` 중복 없음

위반 시 즉시 오류 반환하고 Domain 해 결정 로직은 호출하지 않는다.

### FR-02 빈칸 탐색

빈칸 2개 좌표는 행 우선(row-major)으로 탐색한다.

### FR-03 누락 수 계산

`1..16`에서 현재 격자에 존재하는 수(0 제외)를 제거해 누락된 두 수 `{a,b}`를 구한다.

### FR-04 완성 판정

`0`이 없는 완전 격자에 대해 아래 10개 합이 모두 34이면 참:

- 4개 행
- 4개 열
- 주대각선
- 부대각선

### FR-05 해 결정 및 반환

- 누락 두 수에 대해 최대 두 번 시도:
  - 시도1: 작은 수를 첫 빈칸, 큰 수를 둘째 빈칸
  - 시도2: 반대로 배치
- 성공 시 `int[6]` 반환
- 두 시도 모두 실패 시 `NO_SOLUTION` 반환
- 호출자가 전달한 `grid`는 변경하지 않는다(NFR-04)

## 5. 오류 계약 (code/message)

문자열은 아래와 정확히 일치해야 한다.

| code | message |
|---|---|
| `INVALID_SIZE` | `Grid must be 4x4.` |
| `INVALID_VALUE_RANGE` | `Each cell must be 0 or 1..16.` |
| `INVALID_ZERO_COUNT` | `There must be exactly two zeros (empty cells).` |
| `DUPLICATE_VALUES` | `Values 1..16 must not duplicate (excluding zeros).` |
| `NO_SOLUTION` | `No placement makes a 4x4 magic square with magic sum 34.` |

검증 우선순위:

`INVALID_SIZE` → `INVALID_VALUE_RANGE` → `INVALID_ZERO_COUNT` → `DUPLICATE_VALUES`

## 6. API 계약

### 6.1 입력

- 타입: `list[list[int]]` (4x4)

### 6.2 출력

- 성공: `list[int]` 길이 6
  - `[r1,c1,n1,r2,c2,n2]` (좌표 1-index)
- 실패: `dict[str, str]`
  - `{"code": "...", "message": "..."}`

## 7. 비기능 요구사항 (NFR)

- NFR-01: Domain 라인 커버리지 >= 95%
- NFR-02: Boundary 라인 커버리지 >= 85%
- NFR-03: 동일 입력에 동일 출력(결정론)
- NFR-04: 입력 `grid` 비변형 보장

## 8. 테스트 전략 (Dual-Track TDD)

### 8.1 Boundary Track

- 계약 오류 코드/문구/우선순위 검증
- 무효 입력 시 Domain 미호출 검증

### 8.2 Domain Track

- 빈칸 탐색, 누락 수 계산, 판정 로직 단위 검증
- 두 배치 시도 로직 검증
- `TQ-NOSOL` 두 시도 모두 실패 자동 증명

### 8.3 E2E

- 정상 해결(`E2E-OK`)
- 해 없음(`E2E-NO`)
- 입력 오류(`E2E-ERR` 계열)
- 결정론/입력 비변형

## 9. 수용 기준 (AC)

- AC-01: 유효하지 않은 입력은 우선순위 규칙대로 오류 반환
- AC-02: 유효 입력에 대해 두 시도 내 해가 있으면 `int[6]` 반환
- AC-03: 두 시도 모두 실패하면 `NO_SOLUTION` 반환
- AC-04: 모든 결과는 결정론적이며 입력은 변경되지 않음
- AC-05: 테스트 통과 + NFR 커버리지 기준 충족

## 10. 구현 산출물(현재 기준)

- 테스트: `tests/{boundary,domain,e2e}`
