## 1. 테스트 조건

### 1.1 테스트 환경

| 구분 | 내용 |
|------|------|
| 언어·런타임 | Python **3.10 이상** |
| 테스트 프레임워크 | **pytest** 8.x 이상 (`pyproject.toml` optional-dependencies `dev`) |
| 운영체제 | Windows 10/11, macOS, Linux (저장소에서 검증된 환경: Windows 10) |
| IDE / 편집기 | 제약 없음 (예: VS Code, Cursor, PyCharm) |
| 실행 | 프로젝트 루트에서 `pytest` (`pythonpath`에 `src` 포함) |

### 1.2 테스트 목적

- Boundary: **FR-01** 계약 및 **§5** `code`/`message`·검증 **우선순위** 준수 여부
- Domain: **FR-02 ~ FR-05** (빈칸 탐색, 누락 수, 10선 합 34 판정, 두 배치 시도, `NO_SOLUTION` 경로)
- E2E: 유효/무효 입력에 대한 **최종 응답 형식**(`list` 길이 6 vs `dict`) 및 **결정론·입력 비변형**(NFR-03, NFR-04)

### 1.3 입력값·대표 시나리오 (예시)

| 구분 | 입력·조건 예 | 기대(요지) |
|------|----------------|------------|
| Boundary | 4×4가 아닌 `grid` | `INVALID_SIZE`, PRD §5 `message` 정확 일치 |
| Boundary | 값이 `0`·`1..16` 범위 밖 | `INVALID_VALUE_RANGE` |
| Boundary | `0` 개수 ≠ 2 | `INVALID_ZERO_COUNT` |
| Boundary | `1..16` 중복(0 제외) | `DUPLICATE_VALUES` |
| Boundary | 복수 규칙 동시 위반 | **우선순위**에 따른 **단일** `code` |
| Domain | 두 배치 모두 실패하는 유효 격자 | 내부 로직상 해 없음 → 상위에서 `NO_SOLUTION` 매핑 |
| E2E | 유효·해 존재 격자 | `list[int]` 길이 6, 좌표 **1-index** |
| E2E | 동일 입력 2회 호출 | 동일 출력 |

(전체 케이스는 `tests/boundary`, `tests/domain`, `tests/e2e` 및 PRD **§8.4**에 정의.)

## 2. 전제 조건

- **권장**: 저장소 루트에서 `python -m venv .venv` 등으로 **가상 환경**을 만든 뒤 활성화하고, 그 안에서 `pip` / `pytest` 를 실행한다(절차: [README](../README.md), 실행 세부: [report/9 §3](../report/9.Test_Case_Specification_Report_2026-04-28.md)).
- 소스·테스트 코드가 **오류 없이** import·수집 가능해야 한다 (`pytest` 수집 단계 통과).
- `pip install -e ".[dev]"` 등으로 **개발 의존성**(최소 **pytest**)이 설치되어 있어야 한다.
- `pyproject.toml`의 `pythonpath = ["src"]`, `testpaths = ["tests"]` 설정이 유효해야 한다.
- PRD **§5** 오류 문자열·**§6** API 계약이 테스트 기대값의 기준이며, 구현과 불일치 시 **PRD 우선**으로 정합성을 맞춘다.

## 3. 성공 / 실패 기준

### 3.1 성공

- `pytest` 실행 시 **등록된 전 테스트가 통과**한다.
- PRD **AC-01 ~ AC-04**를 검증하는 테스트가 Green이며, **AC-05**에 따라 **NFR-01**(Domain 라인 커버리지 ≥ 95%), **NFR-02**(Boundary ≥ 85%)를 측정하는 경우 해당 임계치를 **만족**한다(프로젝트에서 커버리지 게이트를 켠 경우).
- 무효 입력 시 **Domain 핵심 해결 로직이 호출되지 않음**을 검증하는 테스트가 설계대로 통과한다.

### 3.2 실패

- 하나 이상의 테스트가 **예상과 다른 결과**를 낳거나 예외로 종료된다.
- 반환 `code`/`message`가 PRD **§5**와 **문자 단위로 일치하지 않는다**(불필요한 공백·대소문자 포함).
- 성공 경로에서 `list` 길이가 6이 아니거나, 좌표가 **1-index** 범위를 벗어난다.
- 동일 입력에 대해 **서로 다른 결과**가 나온다(결정론 위반).
- 호출 후 인자로 전달한 `grid`가 **변경**된다(비변형 위반).

## 4. 특별하게 필요한 절차

- **결과 기록**: `pytest` 로그·커버리지 리포트(해당 시)를 보관하고, 회귀 시 비교 가능하도록 **커밋 단위** 또는 **CI 아티팩트**와 연계한다.
- **실패 시**: 실패 케이스 ID(테스트 함수명), 입력 fixture, 기대/실제를 정리해 **개발팀(또는 담당자)에 즉시 공유**하고, PRD **FR/AC**와의 매핑을 명시한다.
- **추적성**: 새 테스트 추가 시 PRD **FR·AC·§8.4** 줄과 **TC-ID**(또는 테스트 함수명 규약)를 [6.TODO_MagicSquare_4x4_Checklist.md](./6.TODO_MagicSquare_4x4_Checklist.md) 또는 프로젝트 추적표와 맞춘다.
- **`NO_SOLUTION`**: 입력 검증 실패가 **아니라** 유효 격자에 대한 **해 결정 실패**임을 혼동하지 않도록, Boundary와 Domain 테스트를 **분리**해 유지한다(PRD §5 구분).
